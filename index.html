Integrações CBV com Healthmap

Referência para Migração ao Sistema Tabia
📋 Versão 1.0 📅 28/03/2026 🏥 Unimed POA 🔄 Em elaboração
1

Objetivo

Este documento mapeia todas as integrações existentes entre o sistema CBV (Unimed POA) e a plataforma Healthmap, fornecendo uma base estruturada para a futura migração dessas integrações para o sistema Tabia.

O documento cobre:

  • Inventário completo de APIs Healthmap consumidas pelo Modelo Assistencial
  • Mapeamento por subsistema/módulo
  • Parâmetros, contratos e fluxos de cada integração
  • Endpoints externos identificados (fora do Swagger principal)
  • Considerações e riscos para a migração ao Tabia

Autenticação — Compartilhada por Todos os Subsistemas

Todos os subsistemas utilizam o mesmo mecanismo OAuth2 para obter o token de acesso à API Healthmap.

OperaçãoMétodoEndpointTag Swagger
PostToken – Obter token POST /oauth/token autenticacao-resource
ℹ️ Observação O token Bearer deve ser enviado no header Authorization em todas as requisições. O header KeyDS identifica o ambiente (PRD ou HMG).

7

Padrões Arquiteturais Identificados

#PadrãoDescriçãoImpacto na Migração
7.1 Autenticação OAuth2 Todos os subsistemas compartilham o mesmo endpoint /oauth/token. Header KeyDS distingue os ambientes PRD/HMG. O Tabia precisará oferecer mecanismo equivalente ou será necessário um adapter/gateway de autenticação.
7.2 Versionamento de API Padrão /api/{domínio}/v1/{recurso} em todos os endpoints do Swagger principal. Verificar se o Tabia segue o mesmo padrão ou requer ajuste nas chamadas.
7.3 Paginação Respostas de listagem retornam objeto Page<T> com campos pageSize, offset e pageNumber. Se o Tabia usar paginação por cursor ou modelo diferente, será necessário adaptar a camada de consumo.
7.4 Endpoints Transversais Beneficiário, Risco e Autenticação são utilizados por todos os 6 subsistemas. Alto risco: mudança impacta toda a plataforma simultaneamente. Exige prioridade no planejamento.
7.5 Dois Backends em Produção healthmapv3-api-unimedpa-prd… (principal, Swagger disponível) e unimedpa.healthmap.com.br (complementar, sem Swagger). A migração deve cobrir ambos os backends. O complementar exige levantamento adicional.
8

Considerações e Riscos para Migração ao Tabia

8.1 Riscos Identificados
Risco 1 — Endpoints sem Swagger
Risco
4 endpoints em unimedpa.healthmap.com.br sem contrato formal. Dificulta mapear equivalentes no Tabia.
Mitigação
Solicitar documentação ao time Healthmap antes do início da migração.
Risco 2 — Alta Transversalidade
Risco
Beneficiário e Autenticação usados por todos os 6 subsistemas. Incompatibilidade no Tabia afeta toda a plataforma.
Mitigação
Priorizar validação desses contratos no início do projeto de migração.
Risco 3 — Ambientes PRD vs HMG
Risco
URL de homologação usa subdomínio diferente. O Tabia pode ter estratégia diferente de ambientes.
Mitigação
Mapear estratégia de ambientes no Tabia e ajustar variáveis de configuração.
Risco 4 — Modelos de Dados Distintos
Risco
DTOs Healthmap (Usuario, AcaoPlanoAcaoDTO, etc.) podem ter estrutura diferente no Tabia. Necessidade de transformação de dados.
Mitigação
Mapear campos equivalentes (field mapping) antes da implementação.
8.2 Recomendações para o Projeto de Migração
  1. Levantar contratos completos dos 4 endpoints externos (seção 6) junto ao time Healthmap.
  2. Iniciar validação dos contratos do Tabia pelos endpoints transversais: Autenticação e Beneficiário.
  3. Criar camada de abstração (adapter / anti-corruption layer) para isolar as chamadas ao Healthmap, facilitando a substituição pelo Tabia sem impactar regras de negócio.
  4. Definir estratégia de migração: big bang (troca simultânea) vs. strangler fig (migração gradual por subsistema).
  5. Garantir ambiente de homologação do Tabia disponível para testes antes da virada em produção.
  6. Documentar o field mapping entre os modelos de dados Healthmap e Tabia para cada domínio.
